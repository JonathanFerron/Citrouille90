# DxCore Information Relevant to AVR64DU32 (Citrouille90)

## Overview

This document extracts key information from the SpenceKonde/DxCore project that will be useful for developing firmware for the Citrouille90 keyboard using the AVR64DU32 microcontroller.

**DxCore Repository:** https://github.com/SpenceKonde/DxCore

---

## 1. AVR64DU32 Device Characteristics

### Key Features from DxCore Perspective

- **Flash:** 64 KB
- **RAM:** 8 KB SRAM
- **EEPROM:** 512 bytes
- **USERROW:** 512 bytes
- **BOOTROW:** 64 bytes
- **Clock:** 24 MHz maximum
- **USB:** Native USB Full-Speed (the defining feature of DU-series)
- **ADC:** 10-bit
- **No TCD:** Used internally for USB clock generation
- **Timers:**
  - 1× TCA (16-bit, can be split for 6× 8-bit PWM)
  - 2× TCB (16-bit utility timers)
- **Pin Count:** 32-pin variant has 23 usable GPIO

### Comparison to Other Dx-Series

From DxCore's feature table:

| Feature | DA | DB | DD | **DU** |
|---------|----|----|----|----|
| USB | ❌ | ❌ | ❌ | ✅ |
| TCD | ✅ | ✅ | ✅ | ❌ |
| ADC bits | 12 | 12 | 12 | **10** |
| USERROW | 32B | 32B | 32B | **512B** |
| BOOTROW | - | - | - | **64B** |

**Key Insight:** DU-series trades TCD and 12-bit ADC for USB functionality.

## 2. Clock Configuration

### Supported Clock Speeds (DU-series)

From DxCore documentation, **DU-series has limited clock speed options** because USB requires specific frequencies:

**Supported from internal oscillator:**
- 12 MHz (minimum for USB)
- 16 MHz
- 20 MHz
- 24 MHz (maximum rated, recommended for Citrouille90)

**the AVR64DU32 has internal hardware that generates the required 48 MHz USB clock from the PLL.

### Clock Source Configuration

The DU-series can use:
1. **Internal oscillator** (OSCHF) - most common
2. **External clock**
3. **External crystal** (if available on the specific pinout)

**USB Clock Generation:**
- USB requires a 48 MHz reference clock
- On DU-series, this is generated from the internal PLL
- The TCD peripheral (normally available for PWM) is repurposed for USB clock generation
- This is why DU has no TCD available for user applications

### Clock Initialization Pattern

Based on DxCore approach:

```c
// Simplified from DxCore clock initialization
void init_clock(void) {
    // Protected write sequence required for CLKCTRL changes
    _PROTECTED_WRITE(CLKCTRL.OSCHFCTRLA, 
                     CLKCTRL_FRQSEL_20M_gc);  // 20 MHz for USB, but pick 24 instead 
    
    // Wait for clock to stabilize
    while (!(CLKCTRL.MCLKSTATUS & CLKCTRL_OSCHFS_bm));
    
    // Main clock prescaler (typically disabled for max speed)
    _PROTECTED_WRITE(CLKCTRL.MCLKCTRLB, 0);  // No prescaling
}
```

**For Citrouille90:** Any speed from 12-24 MHz works for USB. Recommend 24 MHz for maximum keyboard responsiveness and faster matrix scanning.

## 3. Power Configuration (Matching Spec Section 3)

### DxCore Power Setup Principles

From DxCore documentation, the AVR64DU32 uses **Power Configuration 5b**:

- **VDD:** 5V from USB VBUS
- **VUSB:** 3.3V generated by internal regulator for USB core

### Decoupling Requirements

DxCore follows these guidelines (matches your spec):

```c
// Required capacitors (from DxCore schematics/docs):
// - 4.7µF on VBUS (bulk, X7R/X5R, near USB connector)
// - 2× 100nF on VDD pins (local decoupling)
// - 1× 100nF on VUSB pin (USB core decoupling)
```

**Total capacitance:** 4.7µF + 0.3µF = 5.0µF (safely under USB 10µF limit)

### VUSB Pin Handling

**Critical from DxCore experience:**
- VUSB pin must be decoupled to ground only
- Do NOT connect external loads to VUSB
- VUSB is an OUTPUT from the internal regulator

## 4. Reset and Startup Handling

### Reset Sources (from DxCore Ref_Reset.md)

The AVR64DU32 can reset from:
1. **Power-On Reset (POR)**
2. **Brown-Out Detector (BOD)**
3. **External Reset** (RESET pin)
4. **Watchdog Reset (WDT)**
5. **Software Reset**
6. **UPDI Reset**

### DxCore Reset Flag Handling

**Critical pattern from DxCore:**

```c
// DxCore saves reset flags in GPIOR0 very early (.init3)
// This must be done before clearing RSTCTRL.RSTFR

void init_reset_flags(void) __attribute__((naked)) 
                             __attribute__((used)) 
                             __attribute__((section(".init3")));

void init_reset_flags(void) {
    // Save reset flags to GPIOR0 (user-accessible)
    GPIOR0 = RSTCTRL.RSTFR;
    
    // Clear reset flags (required for proper operation)
    RSTCTRL.RSTFR = RSTCTRL.RSTFR;  // Write 1s to clear
}
```

**Why this matters:**
- Reset flags must be cleared after reading
- If not cleared, certain peripherals may not function correctly
- GPIOR0 preserves the information for user code

### Dirty Reset Detection

DxCore implements **dirty reset detection** to catch software failures:

```c
// In .init3, before main()
if (RSTCTRL.RSTFR == 0) {
    // Dirty reset! Registers weren't cleared
    // Force a clean software reset
    _PROTECTED_WRITE(RSTCTRL.SWRR, 1);
}
```

**For AmberClick90:** Essential for production robustness.

## 6. Timer Configuration

### Available Timers on AVR64DU32

**TCA0 (Type A Timer):**
- 16-bit timer
- Can be split into two 8-bit timers for 6 PWM channels
- DxCore defaults to split mode for `analogWrite()` compatibility

**TCB0 & TCB1 (Type B Timers):**
- 16-bit utility timers
- DxCore typically reserves TCB0 or TCB1 for `millis()`/`micros()`
- Can be used for input capture, one-shot, or 8-bit PWM

**No TCD on DU-series** (used for USB clock generation)

### DxCore Timer Init Pattern

```c
void init_timers(void) {
    // TCA0 in split mode for 6× 8-bit PWM
    TCA0.SPLIT.CTRLD = TCA_SPLIT_SPLITM_bm;  // Enable split mode
    
    // Set period for 8-bit PWM (244 Hz at 20 MHz)
    TCA0.SPLIT.LPER = 0xFF;
    TCA0.SPLIT.HPER = 0xFF;
    
    // Enable timer
    TCA0.SPLIT.CTRLA = TCA_SPLIT_CLKSEL_DIV64_gc | TCA_SPLIT_ENABLE_bm;
}
```

**For AmberClick90:**
- Reserve one TCB for millis() (e.g., TCB1)
- TCB0 available for encoder timing or other uses
- TCA0 available for PWM (LED brightness control)

## 7. Interrupt Handling

### DxCore Interrupt Patterns

From DxCore Ref_Interrupts.md:

**Port Interrupts:**
- All pins can generate interrupts
- Pins 2 and 6 of each port are "async" (can wake from deep sleep)

```c
// Enable pin interrupt (DxCore attachInterrupt style)
void setup_pin_interrupt(void) {
    // Configure pin for interrupt
    PORTA.PIN6CTRL = PORT_PULLUPEN_bm | PORT_ISC_RISING_gc;
    
    // Define ISR
    ISR(PORTA_PORT_vect) {
        // Read interrupt flags
        uint8_t flags = PORTA.INTFLAGS;
        
        // Clear flags by writing 1s
        PORTA.INTFLAGS = flags;
        
        // Handle interrupt
        if (flags & PIN6_bm) {
            // PA6 interrupt occurred
        }
    }
}
```

**For AmberClick90:**
- may Use port interrupts for encoder (PA6, PA7), but not planning on
- Matrix scanning typically polled, not interrupt-driven

### Interrupt Priority

AVR64DU32 supports interrupt priority levels:
- **Level 0:** Normal priority (default)
- **Level 1:** High priority

```c
// Set high priority (from CPUINT peripheral)
CPUINT.LVL1VEC = PORTA_PORT_vect_num;
```

## 8. UPDI Programming Considerations

### DxCore UPDI Handling

**UPDI Pin Protection:**
- UPDI must NEVER be disabled in fuses (no recovery without HV programmer)
- DxCore always keeps UPDI enabled by default

**UPDI vs GPIO:**
- On DU-series, UPDI pin can potentially be configured as GPIO
- Requires HV UPDI for reprogramming if converted to GPIO
- **Not recommended** for production

### SerialUPDI (Recommended Programmer)

DxCore strongly recommends SerialUPDI over jtag2updi:

**Advantages:**
- Faster programming speeds
- More reliable
- Simple hardware (serial adapter + diode)

**Configuration:**
```
Baud rates supported:
- 57600  (slow, for Vcc < 2.7V)
- 230400 (default, works most places)
- 460800 (fast, requires Vcc ≥ 4.5V)
```

## 11. Watchdog Timer (WDT)

### DxCore WDT Patterns

```c
// Enable watchdog with timeout
void wdt_enable(uint8_t period) {
    _PROTECTED_WRITE(WDT.CTRLA, period);
}

// Feed/reset watchdog
void wdt_reset(void) {
    __asm__ __volatile__ ("wdr");
}

// Disable watchdog
void wdt_disable(void) {
    _PROTECTED_WRITE(WDT.CTRLA, 0);
}
```

**Timeout periods:**
```c
WDT_PERIOD_8CLK_gc    // 8 ms
```

## 12. Protected Write Sequence

### Critical for Configuration Registers

Many AVR DU registers require a protected write:

```c
// DxCore implementation
static inline void _PROTECTED_WRITE(volatile uint8_t *reg, uint8_t value) {
    // Disable interrupts
    uint8_t sreg = SREG;
    cli();
    
    // CCP signature for protected write
    CCP = CCP_IOREG_gc;
    
    // Write must occur within 4 cycles
    *reg = value;
    
    // Restore interrupt state
    SREG = sreg;
}
```

**Used for:**
- CLKCTRL configuration
- WDT configuration
- RSTCTRL software reset
- NVMCTRL operations

## 16. ADC Configuration (10-bit on DU)

### DxCore ADC Init Pattern

```c
void init_ADC(void) {
    // Reference voltage (typically VDD)
    ADC0.CTRLC = ADC_REFSEL_VDDREF_gc;
    
    // Sample duration (for higher impedance sources)
    ADC0.SAMPCTRL = 0;  // Default
    
    // Prescaler (ADC clock should be 50-200 kHz)
    ADC0.CTRLC |= ADC_PRESC_DIV128_gc;
    
    // 10-bit resolution
    ADC0.CTRLA = ADC_RESSEL_10BIT_gc;
    
    // Enable ADC
    ADC0.CTRLA |= ADC_ENABLE_bm;
}
```

**Note:** AVR64DU32 has 10-bit ADC, not 12-bit like DA/DB/DD.

## 17. Key Takeaways for AmberClick90

### What to Borrow from DxCore

1. **Reset handling pattern** - Critical for robustness
2. **Clock initialization sequence** - Essential for USB
3. **Protected write macros** - Required for config registers
4. **GPIO initialization patterns** - Clean and documented
5. **Timer setup for millis()** - If using Arduino timing functions
6. **Memory layout conventions** - Flash sections, PROGMEM usage

### What NOT to Use from DxCore

1. **USB stack** - DxCore's USB support is incomplete for DU
2. **Arduino core overhead** - Your bare-metal approach is better
3. **Full Arduino API** - Keep it minimal for keyboard firmware

### Critical DxCore Insights

1. **UPDI must stay enabled** - No recovery without HV programmer
2. **Reset flags must be cleared** - Or peripherals malfunction
3. **USB needs 48 MHz PLL** - Automatically configured on DU
4. **TCD not available** - Used internally for USB clock
5. **10-bit ADC only** - Not 12-bit like other Dx-series
